# 核心抽象

业务层面的关键抽象与领域概念。

## 核心概念定义

- **项目**：代码仓库的映射，记录 Git 平台项目 ID 与仓库地址，是业务单元的代码载体。

- **业务单元**：面向业务的独立交付单元，是租户可独立管理、部署的核心业务载体，关联1个项目（代码仓库）、拥有若干部署计划。

- **部署计划**：完整的配置包，涵盖从代码构建到实例运行的全链路配置，包含 CI 配置、CD 配置和实例配置。

- **实例**：部署计划执行后生成的运行态实体，承载业务单元的运行时状态。（规划中，未实现）

- **依赖**：实例运行所需的中间件与基础能力（如MySQL、Redis、消息队列等），为实例提供运行支撑。（规划中，未实现）

- **资源**：支撑实例运行的基础设施，核心包括CPU、内存、存储、网络。在当前实现中，资源配额嵌入在 K8s 原生工作负载 Spec 中（如 Container.Resources），不作为独立实体。

## 静态结构

```
项目（代码仓库）
    │
    └── 业务单元（1:1 关联项目）
            │
            ├── CI 配置（多个，关联业务单元）
            ├── CD 配置（多个，关联业务单元）
            ├── 实例配置（多个，关联业务单元）
            │
            └── 部署计划（多个）
                    │
                    ├── CI 配置（引用）
                    ├── CD 配置（引用）
                    │
                    └── 实例配置（引用）
                            │
                            └── 依赖绑定（中间件/能力）（规划中）
```

## 动态流转

```
代码仓库 ──提供代码──> 部署计划 ──执行──> 实例（运行态，规划中）
                          │
                          ├── CI 配置 → 构建
                          ├── CD 配置 → 部署
                          └── 实例配置 → 运行
```

## 核心关系说明

1. **项目**是代码仓库映射，每个业务单元关联一个项目；

2. **业务单元**是顶层业务抽象，关联1个项目，拥有多个部署计划和多个配置（CI/CD/实例配置）；

3. **部署计划**是完整配置包，引用 CI 配置（构建）、CD 配置（部署）、实例配置（运行）；

4. **实例**是部署计划执行后的运行态产物，一个部署计划可多次执行生成多个实例（规划中）；

5. 实例运行需绑定**依赖**（中间件能力），一个依赖可被多个实例共享（规划中）。

## 实体关系

| 实体 | 关系 | 目标实体 | 说明 |
|------|------|---------|------|
| 项目 | 1:N | 业务单元 | 一个项目可对应多个业务单元 |
| 业务单元 | 1:1 | 项目 | 每个业务单元关联一个代码仓库 |
| 业务单元 | 1:N | 部署计划 | 一个业务单元可有多个部署计划 |
| 业务单元 | 1:N | CI 配置 | 一个业务单元可有多个 CI 配置 |
| 业务单元 | 1:N | CD 配置 | 一个业务单元可有多个 CD 配置 |
| 业务单元 | 1:N | 实例配置 | 一个业务单元可有多个实例配置 |
| 部署计划 | 1:1 | CI 配置 | 每个部署计划引用一个 CI 配置 |
| 部署计划 | 1:1 | CD 配置 | 每个部署计划引用一个 CD 配置 |
| 部署计划 | 1:1 | 实例配置 | 每个部署计划引用一个实例配置 |
| 实例配置 | N:M | 依赖 | 实例配置可绑定多个依赖，依赖可被共享（规划中） |
| 部署计划 | 1:N | 实例 | 一个部署计划可多次执行生成多个实例（规划中） |
