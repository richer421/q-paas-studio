# Codex Domain Layer Design Paradigms

## Overview

当一个功能域同时包含多层领域语义时，不要先起一个 `Service` 再往里面塞职责。  
应先识别“每层语义到底回答什么问题”，再为每层定义独立、可组合、命名清晰的接口。

核心目标：

1. 一个接口只表达一种稳定语义。
2. 多个小接口按语义组合，不用一个大门面包办所有事情。
3. 门面文件承载公开能力，`model` 承载共享语义。

## 何时使用

当某个 domain 能力同时包含以下两层及以上语义时，使用这套范式：

- 某种规则、结构或流程的定义
- 多个内置方案之间的选择
- 某个动作的实际执行
- 一次运行过程的状态推进与结果收敛

常见危险信号：

- 开始想写一个 `XxxService`
- 一个接口同时承担“定义、选择、执行、状态管理”
- 名字叫 `Manager`，但边界说不清
- 查询、编排、执行、回调全挂在一个门面上

## 语义层拆分

### 1. 定义层

回答“是什么”。

用于表达系统内置的规则、结构、阶段、模板、策略。

典型命名：

- `Spec`
- `Plan`
- `Policy`
- `Template`
- `Pipeline`
- `Strategy`

### 2. 选择层

回答“选哪个”。

用于在多个官方方案、多个内置定义、多个稳定实现之间做选择。

典型命名：

- `Selector`
- `Resolver`
- `Matcher`

### 3. 执行层

回答“怎么执行”。

用于把某个定义或方案真正落到底层依赖、外部系统或具体执行环境。

典型命名：

- `Builder`
- `Renderer`
- `Dispatcher`
- `Executor`

### 4. 生命周期层

回答“这次运行如何从开始走到结束”。

用于统一管理状态推进、结果收敛、回调处理、事件记录。

典型命名：

- `LifecycleManager`
- `Coordinator`
- `StateManager`

## 拆分原则

1. 先按语义层拆，再决定接口名；不要先拍脑袋起一个 `Service`。
2. 只有当某层语义真实稳定存在时，才为它建立独立接口。
3. 查询和简单 CRUD 不要强行塞进这些语义型接口；能在 app 层直接做的就直接做。
4. 接口名要体现职责边界，不要用空泛命名掩盖职责混杂。
5. 一个接口如果同时回答了两个以上不同问题，通常已经拆错了。

## 门面与 model 的放置规则

### 根包门面文件

根包门面文件优先放“这个功能域对外公开的能力接口”。

例如：

- `builder.go`
- `pipeline.go`
- `lifecycle.go`
- `artifact.go`

这些文件适合承载：

- 接口定义
- 配置结构
- 工厂方法
- 类型选择枚举

### model 包

`model` 只放跨实现共享的稳定语义，不放根包对外门面接口。

适合放：

- 输入输出语义对象
- 共享领域实体
- 跨实现复用的依赖端口

不适合放：

- “这个域对外暴露什么能力”的接口定义

## 组合方式

优先采用“语义组合”，而不是“大接口全包”。

推荐关系：

- 定义层描述内容
- 选择层决定采用哪种内容
- 执行层消费定义并执行
- 生命周期层围绕执行过程推进状态

也就是说，应由多个小而稳定的 `xxer` 协作完成能力，而不是由一个大 `Service` 统一包办。

## 中立目录模板

```text
domain/
└── <feature>/
    ├── aliases.go                # 可选：保持根包引用路径稳定
    ├── <role_a>.go               # 根包门面一：接口/配置/工厂
    ├── <role_b>.go               # 根包门面二：接口/配置/工厂
    ├── model/
    │   └── types.go              # 共享语义、领域实体、复用端口
    ├── <impl_a>/
    │   └── <role_a>.go           # role_a 的某种具体实现
    └── <impl_b>/
        └── <role_b>.go           # role_b 的某种具体实现
```

目录说明：

1. `<role_a>.go` / `<role_b>.go`
表示不同语义层的根包门面文件。
2. `model/types.go`
表示跨实现共享的稳定语义层。
3. `<impl_a>/` / `<impl_b>/`
表示按“实现差异”命名的具体实现目录，而不是按临时动作命名。

## 中立示例

假设某个功能域同时包含：

- 一套系统内置规则
- 多个官方可选方案
- 一个具体执行动作
- 一次完整运行过程的状态推进

则更合适的拆法是：

- 一个负责定义规则的接口
- 一个负责选择规则的接口
- 一个负责执行规则的接口
- 一个负责管理运行状态的接口

而不是：

- 一个 `XxxService` 同时负责规则定义、方案选择、动作执行、状态推进和查询读取

## 反例

- 用一个 `Service` 同时承担结构定义、方案选择、执行动作、状态回调
- 用一个 `Manager` 承担所有职责，但名字无法表达边界
- 把根包公开接口塞进 `model`
- 为了“看起来分层”而创建很多没有稳定语义的空接口

## 正例

当多个语义层稳定存在时，为每层建立独立、可组合、命名清晰的接口。  
先明确“这层语义到底回答什么问题”，再决定它应该叫 `Selector`、`Builder`、`Coordinator`，还是别的名字。
